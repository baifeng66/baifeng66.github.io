工作区域和文件状态

工作区域

  • 工作区
    • 工作区是你在电脑上能看到的目录，它包含了实际的项目文件和文件夹。
    • 当你对项目进行修改时，这些修改发生在工作区。
  • 暂存区
    • 暂存区是一个临时存储区域，通常被称为索引（Index）。
    • 在工作区中对文件的修改首先要经过暂存区，你需要将修改的文件添加到暂存区，表示你准备提交这些修改。
    • 通过 git add 命令将工作区的修改添加到暂存区。
  • 本地仓库
    • 本地仓库是 Git 在你电脑上保存项目历史记录的地方。
    • 当你使用 git commit 命令时，Git 会将暂存区的文件快照永久存储到本地仓库中，形成一个新的提交。
    • 通过 git push 命令可以将本地仓库的提交推送到远程仓库。

文件状态

文件有几种不同的状态，这些状态代表了文件在版本控制中的不同阶段。

  1. 未追踪文件（Untracked）：
    • 文件尚未被 Git 追踪，即它在 Git 的版本控制之外。
    • 使用 git add 命令将未追踪文件添加到暂存区，然后再使用 git commit 提交到本地仓库。
  2. 已暂存文件（Staged/Index）：
    • 文件已经添加到 Git 的暂存区，但还未提交到本地仓库。
    • 使用 git add 命令将工作区的修改添加到暂存区。
  3. 已修改文件（Modified）：
    • 文件在工作区中发生了修改，但这些修改尚未被添加到暂存区。
    • 使用 git add 将修改添加到暂存区，然后再使用 git commit 提交到本地仓库。
  4. 未修改文件（Unmodified）：
    • 文件没有在工作区或暂存区中发生修改。
    • 当你的工作区和本地仓库的文件版本一致时，文件处于未修改状态。
  5. 已忽略文件（Ignored）：
    • 通过配置 .gitignore 文件，你可以告诉 Git 忽略某些文件或文件夹，这些被忽略的文件处于已忽略状态。
  6. 已删除文件（Deleted）：
    • 文件已经被删除，但这个删除操作还未被提交到本地仓库。
    • 使用 git rm 命令删除文件，并通过 git commit 提交到本地仓库。
  7. 已提交文件（Committed）：
    • 文件的最新修改已经被提交到本地仓库。
    • 使用 git commit 命令将暂存区的文件提交到本地仓库。

添加和提交文件

查看参考状态

git status

查看仓库提交历史记录

git log

添加到暂存区

git add

  • 可使用通配符 例如：git add *.txt
  • git add . 表示：添加所有文件到暂存区

提交（只提交暂存区中的内容，不会提交工作区中的内容）

git commit -m "注释"

回退版本

git reset --[可选参数]

  • git reset --soft 版本号 : 保存工作区+暂存区
  • git reset --hard 版本号 : 不保存工作区+暂存区
  • git reset --mixed 版本号 : 保存工作区，不保存暂存区

也可以使用 git reset HEAD^ -回退上版本

可使用 git log 和 git reflog去查看（命令操作）版本

查看差异

git diff

  • 比较工作区 暂存区 版本库之间的差异
  • 还可以比较2个特定版本之间的差异
    • 语法：git diff 版本id1 版本id2
    • HEAD -指向最新分支
    • git diff HEAD~ HEAD -比较最新分支和其上一版本差异
    • git diff HEAD^ HEAD -比较最新分支和其上一版本差异
    • git diff HEAD~2 HEAD -比较最新分支和其上二版本差异
    • git diff HEAD~3 HEAD file.txt -比较最新分支和其上三版本中file.txt文件差异

删除文件

rm 文件名

  • 执行此命令后，还得执行git add .去更新暂存区中的数据

git rm 文件名

  • 执行此命令就是直接删除！

忽略文件

.gitignore

  • 系统或者软件自动生成的文件
  • 编译产生的中间文件和结果文件
  • 运行时生成日志文件、缓存文件、临时文件
  • 涉及身份、密码、口令、秘钥等敏感信息文件

> 与merge相比

- merge

- 优点：不会破坏原分支的提交历史，方便回溯和查看
- 缺点：会产生额外的提交节点，分支图比较复杂

- rebase

- 优点：不会新增额外的提交记录，形成线性历史，比较直观和干净
- 缺点：会改变提交历史，改变了当前分支branch out的节点。避免在共享分支使用


## 工作流模型

### Github Flow

GitHub Flow 的主要步骤包括：

1. **创建分支：** 开发新功能或修复 Bug 时，首先从主分支（通常是 `main` 或 `master`）创建一个新分支。
2. **进行修改：** 在新分支上进行代码的开发、修改、测试等操作。每次只专注于一个小的功能或修改，保持代码库的稳定性。
3. **提交更改：** 将修改的代码提交到新分支上。每次提交都应该是一个小而独立的更改，有助于更好地进行代码审查和问题跟踪。
4. **发起 Pull Request：** 当你认为新功能或修改已经完成并通过了本地测试，可以发起一个 Pull Request（PR）。在 PR 中进行代码审查、讨论，确保团队成员都了解和同意你的更改。
5. **合并 Pull Request：** 一旦 PR 经过审查并且通过了测试，就可以将其合并到主分支。这个操作通常由团队的负责人或代码审查者执行。
6. **部署到生产环境：** 一旦更改被合并到主分支，可以将其部署到生产环境。GitHub Flow 鼓励频繁地进行部署，以确保新功能和修改尽早地投入使用。
7. **重复：** 迭代上述步骤，快速响应反馈，持续改进和开发新功能。
