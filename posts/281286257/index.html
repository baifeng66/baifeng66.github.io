目录

HTTP协议

什么是协议？

  • 由其他人或组织负责制定的一套规范或者标准

什么是HTTP协议?

  • w3c制定的一种超文本传输协议（通信协议：发送消息的模板提前被制定好）
  • 什么是超文本？
    • 超文本：非普通文本，流媒体：声音、视频、图片等
    • HTTP协议支持：不仅可以传输普通字符串，同样支持传输声音、视频、图片等
  • 这种协议游走在B/S【B/S结构的系统】之间，B向S发数据【请求(request)】遵循HTTP协议，S向B发数据【响应(response)】同样遵循HTTP协议。
  • HTTP协议包括
    • 请求协议：浏览器 向 web服务器发送数据时，发送的数据需遵循一套规定了发送数据具体格式的标准
    • 响应协议：web服务器 向 浏览器发送数据时，发送的数据需遵循一套规定了发送数据具体格式的标准
  • HTTP协议就是提前制定好的一种消息模板。

HTTP请求协议

  • 请求行
    • 包括三部分
      • 第一部分：请求方式
        • get（常用）
        • post（常用）

        • 什么是URI? 统一资源标识符。代表网络中某个资源的名字，无法定位资源。
        • 什么是URL? 统一资源定位符。代表网络中某个资源的名字，可以定位资源
        • URI和URL什么关系，有什么不同？
          • URL包括URI
          • http://localhost:8080/servlet05/index.html –> URL
          • /servlet05/index.html –> URI
      • 第三部分：HTTP协议版本号
  • 请求头
    • 请求主机
    • 主机端口
    • 浏览器信息
    • 平台信息
    • cookie信息
    • …
  • 空白行
    • 用来区分”请求头“和请求体
  • 请求体
    • 向服务器发送的具体数据
  • HTTP请求协议的具体报文：GET请求
  • HTTP请求协议的具体报文：POST请求

HTTP响应协议

  • 状态行

    • 三部分组成
      • 第一部分：协议版本号（HTTP/1.1）
      • 第二部分：状态码
        • 200 -请求响应成功，正常结束
        • 404 -请求响应失败，资源不存在
        • 405 -请求响应失败，前端请求方式与后端请求处理方式不一致
        • 500 -请求响应失败，服务器端程序出现了异常
        • 4开头的 - 一般是浏览器端的错误
        • 5开头的 - 一般是服务器端的错误
      • 第三部分：状态描述信息
        • ok -正常成功结束
        • not found - 资源找不到

  • 响应头

    • 响应的内容类型
    • 响应的内容长度
    • 响应的时间
    • …

  • 空白行

    • 用来分隔”响应头“和”响应体“

  • 响应体

    • 响应正文，内容是长字符串，会被浏览器渲染，解释并且执行

  • 怎么查看协议内容？

    • 浏览器：F12，看network

  • 怎么向服务器发送GET/POST请求？

    • 默认情况一般全是GET请求
    • 除非特定限制为POST请求，比如表单：method=”post”

  • GET和POST请求有什么区别？

    • get发送的数据会显示在URL上，无法发送大数据量，适合从服务器端获取数据

    • post发送的数据不会显示在URL上（在请求体中），可以发送大数据量，适合向服务器端发送数据

    • get本身是安全的，因为get只是为了从服务器获取数据，不会对服务器造成威胁。

    • post是危险的，因为post是给服务器提交数据。

    • get支持缓存

      • 任何一个get请求的最终的”响应结果“都会被浏览器缓存起来
      • 不希望get走缓存怎么办？
        • 在路径后面添加一个每时每刻在变化的”时间戳“，这样每次请求路径都不一样

    • post不支持缓存

      • 因为服务器的响应结果缓存起来没有意义

什么是模板方法设计模式？

  • 模板方法设计模式是一种行为设计模式，它定义了一个算法的骨架，将一些步骤的实现延迟到子类。
  • 这种模式允许子类重新定义算法的某些步骤而不改变算法的结构。
  • 在模板方法模式中，通常有一个抽象类【模板类】定义了算法的骨架，其中包含了一些抽象方法【模板方法】和具体方法。
  • 子类可以通过实现抽象方法来定制算法的某些步骤，但算法的结构仍然由抽象类控制。
  • 这种模式常用于框架设计，例如在编写具有相似行为的类时，可以使用模板方法模式来避免重复代码。

如何完成资源跳转

在一个web应用中有2种方式，可以完成资源的跳转

  • 转发
  • 重定向

转发和重定向有什么区别？

  • 代码上

    • 转发
    • 重定向

  • 形式上

    • 转发（一次请求）

      • 转发之前之后，浏览器地址栏上的网址不变

    • 重定向（两次请求）

      • 转发之前之后，浏览器地址栏上的网址会有变化

  • 本质区别

    • 转发：由web服务器来控制的。A资源跳转到B资源，这个动作就是由Tomcat服务器内部完成
    • 重定向：由浏览器完成，具体跳转到哪个资源，是浏览器说了算

转发和重定向如何选择？

  • 如果在上一个Servlet当中向request域当中绑定了数据，并且希望从下一个Servlet当中把request域里面的数据取出，使用转发
  • 剩下所有的请求均使用重定向

跳转的下一资源必须是一个Servlet吗？

  • 不一定，可以为任何服务器内部合法的资源(jsp,html,Servlet)

转发会存在浏览器的刷新问题

  • 比如在实现一个新增请求时，在保存成功跳转到成功页面之后，浏览器按刷新几次，就会向数据库重复提交该新增请求（同样的数据）
  • 使用重定向就不会

过滤器 Filter

当前的OA项目存在什么问题？

  • Dept Emp Order 这些Servlet在处理自己的业务之前，都必须进行登录，在此之前所有的Servlet都必须进行一段是否登录的认证，代码太重复了
  • 并且这些Servlet也可能都需要解决乱码问题
  • 使用Servlet中的规范Filter去解决这个问题

Filter是什么？有什么用？执行原理是什么？

  • 是过滤器
  • 可以在Servlet这个目标程序执行之前添加代码，也可以在目标Servlet执行之后添加代码，之前之后都可以添加过滤规则
  • 一般情况，都是在过滤器当中编写公共代码

实现原理

一个过滤器怎么写？

  • 第一步：编写一个Java类实现javax.servlet.Filter，并且实现这个接口中的所有方法
    • init方法：在Filter对象被第一次被创建之后调用，并且只调用一次
    • doFilter方法：只要用户发送一次请求，则执行一次，发送N次请求，则执行N次，在此方法中编写过滤规则
    • destroy方法：在Filter对象被释放/销毁之后调用，并且只调用一次
  • 第二步：在web.xml文件对这个Filter进行配置或者使用注解@WebFilter({“*.do”})

注意：

  • Servlet对象默认情况下，在服务器启动不会自动创建对象
  • Filter对象默认情况下，在服务器启动会自动创建对象
  • 它们都是单例的

目前servlet是否执行，取决于2个条件

  • 在过滤器当中，是否编写了：chain.doFilter(request,response);
    • chain.doFilter(request,response);作用：执行下一个过滤器，如果下面没有过滤器了，执行最终的Servlet
  • 用户发送的请求路径是否和Servlet的请求路径一致
  • 注意：Filter的优先级一定比Servlet的优先级高
    • 如果filter和servlet的请求路径一致，那么一定是先执行filter，接着执行servlet

配置路径

  • /a.do、/b.do、/dept/save 精确匹配
  • /* 匹配所有路径
  • *.do 后缀匹配，不要以 /开始
  • /dept/* 前缀匹配

在web.xml文件中配置Filter文件的时候：执行顺序靠啥？

  • 依靠filter-mapping标签的配置位置，越靠上优先级越高

过滤器的调用顺序：遵循栈的数据结构

使用注解@WebFilter的执行顺序：根据过滤器类名排序

生命周期

  • 和servlet对象生命周期一致
  • 唯一区别：filter默认情况下，在服务器启动阶段就实例化，servlet不会

Filter中有一个责任链设计模式

  • 在程序编译阶段不会确定调用顺序，因为Filter的调用顺序是配置到web.xml文件中的，只需修改web.xml配置文件中的filter-mapping的顺序即可调整filter的执行顺序,显然Filter的执行顺序是在程序运行阶段动态组合的
  • 最大核心思想：动态组合程序的调用顺序

Listener监听器

什么是监听器？

  • 是Servlet规范中的一员
  • 所有监听器接口都以Listener结尾

监听器有什么用？

  • 特殊的时刻如果想执行这段代码，需要想到使用对应的监听器

Servlet规范中提供了哪些监听器？

  • javax.servlet包下：
    • ServletContextListener
    • ServletContextAttributeListener
    • ServletRequestListener
    • ServletRequestAttributeListener
  • javax.servlet.http包下:
    • HttpSessionListener
    • HttpSessionAttributeListener
      • 需要使用@WebListener进行标注
      • 负责监听session域中数据的变化，只要数据变化，则执行对应的方法，主要监测点在session域对象上
    • HttpSessionBindingListener
      • 不需要使用@WebListener进行标注
      • 假设User类实现了该监听器，那么User对象在被放入session的时候触发bind事件，User对象从session中移除的时候触发ubind事件
    • HttpSessionIdListener
      • session的id发送改变的时候，监听器中的唯一一个方法就会被调用
    • HttpSessionActivationListener
      • 监听session对象的钝化和活化
      • 钝化：session对象从内存存储到硬盘文件
      • 活化：从硬盘文件把session恢复到内存

实现一个监听器的步骤：以ServletContextListener为例

  • 第一步：编写一个实现ServletContextListener接口的类，并且实现里面的方法
  • 第二步：在web.xml对ServletContextListener进行配置或者使用注解：@WebListener

注意：所有监听器的方法都是不需要Javaweb程序员调用的，由服务器来负责调用，在某个特殊的事件发生，被服务器调用
