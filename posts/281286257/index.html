Tomcat服务器

目录

  • bin：存放Tomcat服务器的命令文件目录，比如：启动/关闭Tomcat
  • conf：存放Tomcat服务器的配置文件目录，比如：server.xml可以设置端口号
  • lib：存放Tomcat服务器的核心程序目录，因为Tomcat是Java语言编写的，这里面的jar包都是class文件
  • logs：存放Tomcat服务器的日志目录，Tomcat服务器启动等信息都会在此目录生成日志文件
  • temp：存放Tomcat服务器的临时目录，存储临时文件
  • webapps：存放大量的webapp（web applicastion：web应用）
  • work：存放JSP翻译之后的Java文件以及编译之后的class文件

BS结构系统的角色和协议

对于一个动态的web应用（页面中数据是动态的，根据数据库中数据的变化而变化）来说，一个请求和响应的过程有多少个角色参与，角色和角色之间有多少个协议

  • 有哪些角色
    • 浏览器软件的开发团队（浏览器：谷歌、360…）
    • Web server 的开发团队（web server：Tomcat、Jetty…）
    • DB Server的开发团队（DB server：Mysql、Oracle…）
    • webapp的开发团队（web应用是程序员开发的）
  • 角色和角色之间需要遵循哪些规范/协议
    • Webapp开发团队 和 web server开发团队之间有一套规范：Servlet规范
      • Servlet规范作用：
        • Webapp 和 web server 解耦合
    • Browser 和 Web server之间有一套传输协议：HTTPS协议
    • Webapp开发团队 和 DB server开发团队之间有一套规范：JDBC规范

  • 第四步：在WEB-INF目录下新建一个目录：lib
    • 必须全部小写【Servlet规范】，此目录下一定存放webapp需要的第三方jar包
  • 第五步：在WEB-INF目录下新建一个文件：web.xml
    • 必须全部小写【Servlet规范】，此配置文件描述了请求路径和Servlet类之间的对照关系

  • 第六步：编写一个Java程序，必须实现Servlet接口

    • 这个Servlet接口不在JDK中（因为Servlet属于JavaEE,是另外一套类库）
    • 注意：编写程序时，Java源代码随便放哪里，只需要把编译之后的class文件放到classes目录下面即可

  • 第七步：编译我们写的java程序

    • 重点：发现编译不过！咋办：配置环境变量CLASSPATH

      export CLASSPATH=$CLASSPATH:${TOMCAT_HOME}/lib/servlet-api.jar

    • 思考：以上配置的环境变量和Tomcat服务器运行有没有关系？

      • 无关系，以上配置这个环境变量只是为了让我们写的代码正常编译生成class文件

  • 第八步：将以上编译之后的.class文件拷贝到WEB-INF/classes目录下

  • 第九步：在web.xml文件编写配置信息，让 请求路径 和 Servlet类名 关联在一起

    • 在web.xml文件中注册Servlet类
  • 第十步：启动Tomcat服务器
  • 第十一步：打开浏览器，输入url
    • Url: http://localhost:8080/crm/h
    • 重要：浏览器上请求路径不能随便写，必须和web.xml文件中的url-pattern一致

Servlet对象由谁来维护？

  • Servlet对象的创建、销毁，对象上方法的调用，程序员无权干涉
  • 由Tomcat服务器【又称Web 容器】全权负责

思考：我们自己new 的Servlet对象受Web容器的管理吗？

  • 不受的
  • Web容器创建的Servlet对象会被放到一个集合【HashMap】当中,只有放到这个集合中的Servlet才能被Web容器管理
  • Web容器底层应该是有HashMap集合，存储了Servlet对象和请求路径之间的关系

研究：服务器在启动的时候，Servlet对象有没有被创建出来？

    • 经过测试：服务器启动时候Servlet并不会实例化
  • 怎么让服务器启动的时候创建Servlet对象勒
    • 在web.xml文件中的servlet标签中添加<load-on-startup>1</load-on-startup>，该标签里面填写整数，整数越小优先级越高

HTTP协议

什么是协议？

  • 由其他人或组织负责制定的一套规范或者标准

什么是HTTP协议?

  • w3c制定的一种超文本传输协议（通信协议：发送消息的模板提前被制定好）
  • 什么是超文本？
    • 超文本：非普通文本，流媒体：声音、视频、图片等
    • HTTP协议支持：不仅可以传输普通字符串，同样支持传输声音、视频、图片等
  • 这种协议游走在B/S【B/S结构的系统】之间，B向S发数据【请求(request)】遵循HTTP协议，S向B发数据【响应(response)】同样遵循HTTP协议。
  • HTTP协议包括
    • 请求协议：浏览器 向 web服务器发送数据时，发送的数据需遵循一套规定了发送数据具体格式的标准
    • 响应协议：web服务器 向 浏览器发送数据时，发送的数据需遵循一套规定了发送数据具体格式的标准
  • HTTP协议就是提前制定好的一种消息模板。

HTTP请求协议

  • 请求行
    • 包括三部分
      • 第一部分：请求方式
        • get（常用）
        • post（常用）
        • delete
        • put
        • head
        • trace
        • Options
      • 第二部分：URI
        • 什么是URI? 统一资源标识符。代表网络中某个资源的名字，无法定位资源。
        • 什么是URL? 统一资源定位符。代表网络中某个资源的名字，可以定位资源
        • URI和URL什么关系，有什么不同？
          • URL包括URI
          • http://localhost:8080/servlet05/index.html –> URL
          • /servlet05/index.html –> URI
      • 第三部分：HTTP协议版本号
  • 请求头
    • 请求主机
    • 主机端口
    • 浏览器信息
    • 平台信息
    • cookie信息
    • …
  • 空白行
    • 用来区分”请求头“和请求体
  • 请求体
    • 向服务器发送的具体数据
  • HTTP请求协议的具体报文：GET请求
  • HTTP请求协议的具体报文：POST请求

HTTP响应协议

  • 状态行

    • 三部分组成
      • 第一部分：协议版本号（HTTP/1.1）
      • 第二部分：状态码
        • 200 -请求响应成功，正常结束
        • 404 -请求响应失败，资源不存在
        • 405 -请求响应失败，前端请求方式与后端请求处理方式不一致
        • 500 -请求响应失败，服务器端程序出现了异常
        • 4开头的 - 一般是浏览器端的错误
        • 5开头的 - 一般是服务器端的错误
      • 第三部分：状态描述信息
        • ok -正常成功结束
        • not found - 资源找不到

  • 响应头

    • 响应的内容类型
    • 响应的内容长度
    • 响应的时间
    • …

  • 空白行

    • 用来分隔”响应头“和”响应体“

  • 响应体

    • 响应正文，内容是长字符串，会被浏览器渲染，解释并且执行

  • 怎么查看协议内容？

    • 浏览器：F12，看network

  • 怎么向服务器发送GET/POST请求？

    • 默认情况一般全是GET请求
    • 除非特定限制为POST请求，比如表单：method=”post”

  • GET和POST请求有什么区别？

    • get发送的数据会显示在URL上，无法发送大数据量，适合从服务器端获取数据

    • post发送的数据不会显示在URL上（在请求体中），可以发送大数据量，适合向服务器端发送数据

    • get本身是安全的，因为get只是为了从服务器获取数据，不会对服务器造成威胁。

    • post是危险的，因为post是给服务器提交数据。

    • get支持缓存

      • 任何一个get请求的最终的”响应结果“都会被浏览器缓存起来
      • 不希望get走缓存怎么办？
        • 在路径后面添加一个每时每刻在变化的”时间戳“，这样每次请求路径都不一样

    • post不支持缓存

      • 因为服务器的响应结果缓存起来没有意义

什么是模板方法设计模式？

  • 模板方法设计模式是一种行为设计模式，它定义了一个算法的骨架，将一些步骤的实现延迟到子类。
  • 这种模式允许子类重新定义算法的某些步骤而不改变算法的结构。
  • 在模板方法模式中，通常有一个抽象类【模板类】定义了算法的骨架，其中包含了一些抽象方法【模板方法】和具体方法。
  • 子类可以通过实现抽象方法来定制算法的某些步骤，但算法的结构仍然由抽象类控制。
  • 这种模式常用于框架设计，例如在编写具有相似行为的类时，可以使用模板方法模式来避免重复代码。

HttpServlet类专门为HTTP协议准备的。比GenericServlet更适合HTTP协议下的开发

目前为止，我们接触了哪些servlet规范中的接口?

  • javax.servlet.Servlet - 核心接口
  • javax.servlet.ServletConfig - 配置信息接口
  • javax.servlet.ServletContext - 上下文接口
  • javax.servlet.ServletRequest - 请求接口
  • javax.servlet.ServletResponse - 响应接口
  • javax.servlet.ServletException - 异常类
  • javax.servlet.GenericServlet - 标准通用的Servlet抽象类

http包下都有哪些类和接口？

  • javax.servlet.http.HttpServlet（HTTP协议专用Servlet类，抽象类）
  • javax.servlet.http.HttpServletRequest（HTTP协议专用请求对象）
  • javax.servlet.http.HttpServletResponse（HTTP协议专用响应对象）

HttpServletRequest对象中封装了什么信息？

  • HttpServletRequest【简称request对象】
  • 封装了请求协议的全部内容
  • Tomcat服务器将”请求协议“中的数据全部解析出来，然后全部封装到request对象当中。

如何完成资源跳转

在一个web应用中有2种方式，可以完成资源的跳转

  • 转发
  • 重定向

转发和重定向有什么区别？

  • 代码上

    • 转发
    • 重定向

  • 形式上

    • 转发（一次请求）

      • 转发之前之后，浏览器地址栏上的网址不变

    • 重定向（两次请求）

      • 转发之前之后，浏览器地址栏上的网址会有变化

  • 本质区别

    • 转发：由web服务器来控制的。A资源跳转到B资源，这个动作就是由Tomcat服务器内部完成
    • 重定向：由浏览器完成，具体跳转到哪个资源，是浏览器说了算
